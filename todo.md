

# clishot 需求文档（迭代方案）

## 1. 背景与目标
**目标**：解决实验报告需要“终端截图”但手工截图不方便、窗口太小截不全、sandbox 终端不可见等问题。  
**核心交付物**：从纯文本终端输出生成一张或多张图片（PNG/JPG），排版像终端，尺寸统一，方便直接插入实验报告。

## 2. 总体原则
- **不做真实截屏**：不依赖 GUI、不依赖终端窗口像素，避免“截不全/不可截图”的根问题。
- **纯文本优先**：只保留文字与换行（可做基础清洗），忽略颜色样式。
- **统一规格**：统一宽度（列宽或像素宽）+ 自动分页（推荐）。
- **CLI 做确定性能力，skill 做编排**：CLI 输出必须稳定可预测；skill 负责交互与环境差异整合。

## 2.1 技术实现方案（我们已明确的路线）
### A. 采集输出（系统能力优先，不绑定 clishot）
- 通过系统重定向/管道把终端输出写入日志文件（纯文本）。
- Windows 优先考虑 PowerShell 的重定向写法；macOS/Linux 用 shell 重定向写法。
- 采集侧只负责“拿到完整文本”，不追求样式与终端窗口一致。

### B. MVP 核心（clishot CLI）：从文本/日志自动生成图片
- 输入：读取日志文件或从 stdin 接收文本。
- 处理：只保留文字与换行，做基础清洗（不可见控制字符、`\r` 覆盖行/进度条的兜底策略）。
- 输出：生成统一尺寸的 PNG/JPG；长内容自动分页为多张图片。
- 统一规格策略：固定列宽（80/100/120）或固定像素宽（A4/1200px）；二选一作为默认策略。
- V1 不依赖前端页面渲染/截图引擎：优先采用 Node.js 直接绘制文本到图片（等宽字体、行距、边距、分页）。

### C. 非 MVP（后续增强）：前端渲染“伪终端截图” + 截图引擎
- 思路：生成一个 HTML/CSS 的终端风格界面（标题栏/圆角/阴影/字体/行距），把纯文本塞进 `<pre>` 进行排版。
- 截图：用类似 Playwright 的方式对该页面截图输出 PNG，从而得到更像终端窗口的“假截图”。
- 主题/风格：做成可配置模板（mac 风、Windows Terminal 风、极简风、报告风等），由 skill 负责询问/选择。

### D. 备选渲染路线（可选）
- 不走前端截图时，也可以用 Node.js 直接绘制（canvas/图形库）将文本渲染为图片；优点是链路更轻，缺点是字体与排版一致性需要额外控制。

## 3. 功能边界（明确不做什么）
**不做（至少 V1 不做）**
- ANSI 颜色/粗体/下划线等样式还原
- 复杂 TUI 界面“动态刷新”的高保真复现（如 `htop`、进度条反复刷新）
- 真正的桌面窗口截图（系统截屏 API）

**做**
- 文本输入（stdin/文件）→ 统一排版 → 图片输出（支持分页）
- 可选终端风格外框（标题栏/圆角/阴影）作为“伪截图”效果

## 4. 迭代规划

### V1（MVP）：Render-only（最稳、最通用）
**目标**：只要拿到文本，就一定能生成可贴报告的图片。

**CLI 能力（P0）**
- 输入：
  - `stdin`：支持管道/重定向输入
  - `--in <file>`：从日志文件读入
- 输出：
  - `--out <path>`：输出单张图片或多张分页图片（例如 `out-001.png`、`out-002.png`）
- 排版：
  - 等宽字体渲染
  - 保留换行
  - 处理常见控制字符：至少清理不可见控制字符；对 `\r`（回车覆盖行）给出简单策略（例如转换为换行或保留最后一次覆盖结果）
  - 自动换行（按列宽/像素宽策略之一）
  - 超长内容自动分页（推荐默认开启）
- 主题（P0/P1 之间可调整）：
  - 至少 1 个默认主题（深色“终端风”或极简白底代码块风）
  - 可选 `--theme` 切换（哪怕只有 `terminal` / `paper` 两种）

**CLI 不做（V1 明确不做）**
- 执行命令并捕获输出（可留到 V2，或交给 skill/系统命令处理）
- 颜色与 ANSI 样式解析

**skill 在 V1 做什么**
- 负责“采集输出”的整合：
  - 在本机：引导/生成重定向命令（PowerShell / bash）将输出保存为文本，再调用 `clishot` 渲染
  - 在 sandbox：直接拿到 stdout 文本，传给 `clishot` 的 stdin 渲染
- 负责交互参数：
  - 询问或默认：主题、宽度（80列/120列 或 A4 宽）、是否分页、标题
- 负责回传：
  - 返回生成图片路径，便于插入报告

---

### V2：Convenience + 风格增强
**目标**：更省事、更像“终端截图”，但仍保持纯文本路线。

**CLI 增强（P1）**
- 增加 `run` 子命令（可选）：
  - `clishot run -- <cmd...>`：执行命令并捕获 stdout/stderr，再直接渲染成图
  - 是否加入取决于：你是否希望 CLI 单独使用时“一条命令出图”
- 风格增强：
  - 更像终端窗口的标题栏（可显示：命令、时间、cwd、hostname 的可选字段）
  - 多主题（mac 风 / windows terminal 风 / minimal / paper）
- 更强文本规整：
  - tab 展开策略
  - 可选行号
  - 可选裁剪过长行（通常不推荐，报告更需要完整）

**skill 在 V2 增强**
- 根据场景自动选择：
  - 直接调用 `clishot run`（当 agent 可执行命令）
  - 或 `stdout → clishot render`（当执行在不可见环境）
- 提供“实验报告模式”对话：
  - 自动生成标题（实验名/步骤/命令）
  - 输出图片命名按步骤编号

## 5. 兼容性说明（Win / macOS / Linux；终端差异）

### 操作系统
- **Windows（优先，PowerShell）**
  - 采集日志：PowerShell 重定向语法与 bash 不同，建议由 skill 统一生成命令或直接走 stdin。
  - 字体：建议默认使用 Windows 常见等宽字体（如 Cascadia Mono/Consolas），但为了结果一致性，更推荐打包一份开源等宽字体随 CLI 分发。
- **macOS / Linux（后续支持）**
  - 采集日志：bash/zsh 常用 `> out.log 2>&1`，同样可由 skill 统一封装。
  - 渲染：如果使用内置字体会有平台差异，打包字体可最大化一致。

### “不同终端是否有区别？”
- 你的方案核心是**不依赖终端模拟器**，所以“Windows Terminal / iTerm2 / VSCode Terminal”本身差异不影响最终渲染。
- 影响最大的其实是“输出文本里是否带控制符/进度条刷新/TUI”：
  - V1 只保证对普通命令输出（编译日志、`ls/dir`、脚本输出）效果最好
  - 对 TUI/动态刷新输出，V1 只能做到“尽量可读”，不追求复刻界面

## 6. 测试策略（你后续实现时怎么验证）

**单元测试（P0）**
- 文本清洗函数：
  - 控制字符清理（不可见字符）
  - `\r` 行覆盖策略
  - 换行规范化（`\r\n`/`\n`）
- 换行与分页：
  - 给定列宽/像素宽，验证分行结果
  - 给定页面高度，验证分页数量与每页行数

**Golden Test（P0/P1）**
- 固定输入文本 + 固定主题/宽度/字体 → 输出 PNG
- 用“像素级对比”或“结构性对比（尺寸/页数/文本位置）”验证渲染稳定性  
  - 建议优先做：尺寸、页数、关键行是否在对应页（比像素级更抗平台差异）
  - 如果你打包字体并固定渲染引擎，像素级对比也可行

**CLI E2E 测试（P0）**
- `echo`/简单脚本输出 → 管道给 `clishot` → 生成文件存在且尺寸正确
- 读取 `--in` 文件 → 输出分页文件命名正确

**跨平台 CI（P1）**
- Windows first；后续加 macOS/Linux
- 关键点：确保字体一致、渲染引擎一致，否则 Golden 容易漂移

## 7. 命名与可变更性
- 你现在仓库/目录叫 `clishot` 没问题。
- 将来发布 npm 时：
  - 如果 `clishot` 包名被占用：用 `@scope/clishot` 仍可让命令名叫 `clishot`
- 以后改名是可行的：
  - 改仓库名最容易
  - 改命令名与包名可通过“保留旧命令别名/旧包提示迁移”平滑过渡
